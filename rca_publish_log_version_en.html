RCA: publish_log_version Ignores load_ids — Schema Change Deadlock

ClassificationBugFix / Shared-data Lake / Transaction Visibility / Schema Change Deadlock
Affected modulesbe/src/storage/lake/transactions.cpp (publish_log_version)
Trigger(1) Table has an active Schema Change (shadow index exists); (2) a transaction is committed via SQL BEGIN; INSERT ...; COMMIT; during that window
ImpactThe explicit-transaction INSERT stays in COMMITTED forever; the Schema Change job stalls in FINISHED_REWRITING and never completes; subsequent transactions on the same partition cannot become visible
RecoveryOnly by issuing CANCEL ALTER TABLE
Data safetyNo data loss (4-segment .log files exist on shared storage), but the partition's visible version is frozen and new data remains invisible to queries until cancellation

1. Summary

During a Schema Change, FE calls publishLogVersionBatch so BE can materialize each transaction's .log on shadow-index tablets into a versioned .vlog. The BE function publish_log_version() always builds the 2-segment path (no load_id suffix) to locate the source .log. For transactions committed via SQL BEGIN; INSERT ...; COMMIT;, the writer places the log at the 4-segment path (with load_id suffix). BE always returns NotFound, publishPartition catches the exception and returns false, and the transaction stays in COMMITTED forever.

More importantly, because the transaction never becomes VISIBLE, the partition's visibleVersion is permanently frozen, and the Schema Change job — which must wait for visibleVersion == commitVersion - 1 in FINISHED_REWRITING — can never finish. This forms a "transaction-waits-on-alter, alter-waits-on-transaction" deadlock. The only way out is CANCEL ALTER TABLE.

6. Who Populates load_ids

TxnInfoPB.load_ids has exactly one writer in FE — TxnInfoHelper.java:41, gated by a strict condition:

if (state.getLoadIds() != null
        && state.getSourceType() == TransactionState.LoadJobSourceType.INSERT_STREAMING) {
    infoPB.setLoadIds(...);
}

TransactionState.loadIds is set exclusively by TransactionStmtExecutor.commitStmt. Conclusion: load_ids is non-empty iff the transaction was committed through SQL BEGIN; ... COMMIT;. HTTP _stream_load and HTTP multi-statement transaction load do not set the field and do not trigger the bug.

8. Schema Change Deadlock Mechanism (Critical)

This is not "stuck until the alter finishes" — the alter itself never finishes and requires manual intervention.
A single explicit transaction (with non-empty load_ids) committed while the shadow index exists is enough to wedge the alter in the FINISHED_REWRITING phase indefinitely.

8.1 The deadlock loop

  1. Schema Change creates the shadow index (IndexState.SHADOW) and enters the backfill phase;
  2. A client issues BEGIN; INSERT INTO ... ; COMMIT; during the alter window; the resulting transaction has non-empty TxnInfoPB.load_ids;
  3. The transaction's .log files are written by BE to the 4-segment path on both normal and shadow tablets;
  4. FE's PublishVersionDaemon invokes publishPartition, which for this partition needs to perform both:
    • Utils.publishLogVersion(shadowTablets, ...) → BE publish_log_version() → searches the 2-segment path → NotFound → throws
    • Utils.publishVersion(normalTablets, ...) → would succeed, but is never executed because the previous call throws first
  5. publishPartition enters its catch block, calls setErrorMsg, and returns false (PublishVersionDaemon.java:1099-1110);
  6. The partition's publish for this commit version did not succeed → the transaction's commitVersion never becomes visibleVersion on this partition;
  7. The transaction remains in COMMITTED indefinitely;
  8. In the FINISHED_REWRITING phase, the Schema Change calls readyToPublishVersion(), which at LakeTableSchemaChangeJob.java:813 checks commitVersion == physicalPartition.getVisibleVersion() + 1; since visibleVersion no longer advances, this check never succeeds → alter loops here forever;
  9. The shadow index remains in SHADOW → publishLogVersion keeps being called → keeps failing → back to step 4 — infinite loop.

9. Only Exit Path

There is no automatic recovery. The idempotency fallback relies on the target .vlog already existing; but the .vlog is never produced (the source 2-segment .log never existed in the first place). Retries return NotFound until the deadlock is broken externally.

The only operator action that ends the loop:

CANCEL ALTER TABLE <table_name> COLUMN;

After cancellation:

  • The shadow index is dropped, so the shadowTablets set becomes empty;
  • publishPartition no longer calls publishLogVersion; publishVersion (normal tablets) runs and succeeds;
  • Backed-up COMMITTED transactions transition to VISIBLE on the next publish daemon tick;
  • The user must re-issue the Schema Change during a window when no explicit transactions overlap.

10. Impact

  • Once triggered, the Schema Change stalls indefinitely in FINISHED_REWRITING and the alter job will not complete on its own.
  • The triggering explicit-transaction INSERT remains in COMMITTED indefinitely.
  • Subsequent stream loads / INSERTs on the same partition also fail to become visible, because every publishPartition on that partition has to pass through the shadow tablet path and throws.
  • Clients hit txn_visible_wait_timeout and return errors.
  • The publish daemon retries continuously, generating log noise and consuming threads, potentially slowing publishes on unrelated partitions on the same node.
  • No data loss: source .log files and segments remain intact on shared storage; after cancellation, the normal-path publish catches everything up.
