1. Verdict (one-line)

Not caused by customer workload. This is an internal StarRocks 4.1.0 BE bug — publish_log_version() is missing its load_ids branch, so when a table is undergoing a Schema Change that requires real data backfill (in this case ADD COLUMN merchantId ... AS COALESCE(...), a virtual/generated column) and a client commits via SQL explicit transactions concurrently, BE looks for the source .log at the wrong path; transactions stay in COMMITTED forever and the alter job is wedged in FINISHED_REWRITING. CANCEL ALTER TABLE is rejected by code in this state; the customer was forced to recover via DROP TABLE ... FORCE, which permanently lost the writes that committed but never became visible during the outage window.

3. Causal chain (internal anomaly → customer-visible symptom)

  1. Precondition: Finaloop runs ADD COLUMN merchantId ... AS COALESCE(...) on the Primary-Key table businessEvents.businessEventActions (monthly partitioned by date_trunc('month', eventTime), 6 buckets) on production 4.1.0. Because this is a generated column, it cannot take the FSE v2 fast path and must do a real data backfill; the alter enters RUNNING and creates the shadow index (IndexState.SHADOW).
  2. Continued writes: the customer's stream-load pipeline (which under the hood uses the explicit-transaction path, evidenced by InsertTxnCommitAttachment) keeps writing to the table, populating TxnInfoPB.load_ids.
  3. BE write: delta_writer calls TabletManager::put_txn_log(); because has_load_id() is true, the .log lands at the 4-segment path with the load_id suffix (tablet_manager.cpp:907).
  4. FE publish: PublishVersionDaemon.publishPartition splits tablets by MaterializedIndex.IndexState — normal tablets go through publishVersion, shadow tablets through publishLogVersion (PublishVersionDaemon.java:1062-1083).
  5. BE read bug: publish_log_version() only branches on combined_txn_log; there is no load_ids_size() > 0 branch. It calls fs::copy_file() on the 2-segment path → NotFound (transactions.cpp:507-542). The idempotency fallback requires the target .vlog to already exist, but it never has.
  6. Exception swallowing: publishPartition wraps publishLogVersion (shadow) and publishVersion (normal) in the same try block; when the former throws, the latter never runs, and the catch returns false (PublishVersionDaemon.java:1081-1110).
  7. Transaction frozen: the partition's publish failed → the transaction's commitVersion never becomes visibleVersion → the transaction remains in COMMITTED forever.
  8. Alter deadlock: LakeTableSchemaChangeJob.readyToPublishVersion() requires commitVersion == visibleVersion + 1 to advance to FINISHED; the condition is never satisfied (LakeTableSchemaChangeJob.java:805-820) → the alter loops in FINISHED_REWRITING forever. The customer noticed the symptom via the "future" FinishTime in SHOW ALTER.
  9. Recovery blocked: at 18:30 UTC the customer ran CANCEL ALTER TABLE; cancelImpl's guard at line 1166 (jobState == FINISHED_REWRITING && tableExists()) rejected it with "Job can not be cancelled. State: FINISHED_REWRITING".
  10. Customer-visible symptoms:
    • Stream load clients timed out at ~90s with Wait txn state timeout 89957000 us;
    • Subsequent stream loads / INSERTs on the same partition were all stuck in COMMITTED; new data in the entire table was invisible;
    • SHOW ALTER TABLE COLUMN showed the job permanently stuck in FINISHED_REWRITING;
    • Finaloop's product could not ingest new business events for 5.5 hours, generating "a flood of complaints from our customers".

4. Contributing factors

  • Incorrect prior guidance on this ALTER's safety (acknowledged by Kevin Chen at the end of the thread): a week before the incident the customer asked whether this ALTER was safe, and was told "background backfill, no write lock". In reality, virtual/generated columns require a real data backfill, exactly hitting this bug. The customer therefore executed the ALTER during a high-traffic window with no backup plan.
  • Read/write routing asymmetry: the writer put_txn_log routes on has_load_id() (2 branches); the reader load_txn_log() has all three branches; but publish_log_version() only has two — the same routing protocol is reimplemented in three places and one of them drifted.
  • Exception bundling: publishPartition wraps shadow and normal publishes in the same try block; an exception from the shadow side blocks the normal side from advancing visibility, turning a localized failure into a whole-partition outage.
  • Cancel-during-FINISHED_REWRITING is forbidden by design: this guard protects data integrity, but in this bug it locks operators out, and there is no FORCE / break-glass bypass.
  • Idempotency fallback is unusable in "never succeeded" cases: the existing fallback only short-circuits when the .vlog already exists. It cannot recognize a persistent "source path is wrong" failure.
  • Mis-diagnosis: cache: the on-call initially suspected a cache issue and disabled caches, wasting ~10 minutes before Chun Zhao noticed "one tablet block the whole ingestion" and the investigation got back on track.
  • Mixed customer pipeline: Finaloop writes via both stream load API and SQL; on the surface the failure looks like a stream-load issue, but underneath it goes through the explicit-transaction path. Disambiguating requires checking InsertTxnCommitAttachment (vs StreamLoadTxnCommitAttachment) and the origin of loadIds (only TransactionStmtExecutor.commitStmt writes it).

5. Why X failed — why the usual safety nets did not catch this

  • FE retries: FE classifies NotFound as retryable, but every retry hits the same dead path, with no upper bound on retries.
  • Idempotency probe: on NotFound, publish_log_version probes whether the target .vlog already exists; but the .vlog was never generated (the 2-segment source .log never existed), the probe always misses.
  • Alter watershedTxnId barrier: the alter's WAITING_TXN phase waits for all txns with txn_id < watershedTxnId to finish, but this bug's triggering transactions are committed after the alter started, so the barrier doesn't protect them.
  • CANCEL ALTER TABLE: refuses to cancel in FINISHED_REWRITING; no FORCE override.
  • Dashboards / alerts: CPU/Mem/QPS all green (resources were idle); publish_log_version WARNINGs have no dedicated metric; you have to read logs or alter job state to spot the issue.
  • Pre-incident safety advice: StarRocks/CelerData side gave an explicit "this ALTER is safe" answer a week prior, with no warning about the backfill path.

6. Recommended actions (in priority order)

Step 1 — Fix the root cause: BE hotfix for publish_log_version()

Add a txn_info.load_ids_size() > 0 branch to publish_log_version() in be/src/storage/lake/transactions.cpp:507, mirroring the three-branch structure of load_txn_log() (same file, line 123): read each per-load_id .log from the 4-segment path, merge them into a single TxnLog, write one .vlog, queue source files for async deletion.

// Fix sketch (as the first branch of the for-loop)
if (txn_infos[i].load_ids_size() > 0) {
    ASSIGN_OR_RETURN(auto per_tablet_logs,
                     load_txn_log(tablet_mgr, {tablet_id}, txn_infos[i]));
    auto& logs = per_tablet_logs[0];
    if (logs.empty()) {
        auto vlog_path = tablet_mgr->txn_vlog_location(tablet_id, log_versions[i]);
        ASSIGN_OR_RETURN(auto fs, FileSystemFactory::CreateSharedFromString(vlog_path));
        if (fs->path_exists(vlog_path).ok()) continue;  // idempotent
        return Status::NotFound(...);
    }
    auto merged = merge_txn_logs(logs);
    RETURN_IF_ERROR(tablet_mgr->put_txn_vlog(merged, log_versions[i]));
    for (const auto& load_id : txn_infos[i].load_ids()) {
        files_to_delete.emplace_back(
            tablet_mgr->txn_log_location(tablet_id, txn_infos[i].txn_id(), load_id));
    }
    continue;
}

Expected effect: once the BE binary is replaced and rolling-restarted, the next publish scheduler tick (~1s by default) will successfully produce the .vlog and the partition's visibleVersion will advance to commitVersion; on the following LakeTableSchemaChangeJob tick, the alter transitions to FINISHED and the shadow index is promoted. No data loss.

Step 2 — Customer emergency path (what Finaloop actually did)

Without a hotfix available, Finaloop followed this procedure. Document it as the reference path for any future incident where a hot patch is not yet ready:

  1. Create businessEventActions_v2 with the same schema, including the new generated column;
  2. submit task re_ingest_business_events as INSERT INTO businessEventActions_v2 (...) SELECT ... FROM businessEventActions; — copy currently visible data to the new table;
  3. DROP TABLE businessEventActions FORCE; (plain DROP is rejected with "There are still some transactions in the COMMITTED state"; FORCE is required);
  4. ALTER TABLE businessEventActions_v2 RENAME TO businessEventActions;;
  5. Re-grant all ROLE / USER permissions on the new table (SHOW GRANTS FOR <user> → manual GRANT);
  6. Resume ingestion; warm up PK index / SST caches (the customer used cache select ... from to proactively warm).
Data-loss warning: the SELECT ... FROM businessEventActions in step 2 only sees data below visibleVersion. Any write that was COMMITTED but never VISIBLE during the outage window (5.5 hours of business events) was permanently lost when DROP TABLE ... FORCE executed. Finaloop absorbed this data loss in this incident.

Step 3 — Long-term hardening

  • Extract the "has_load_id() / combined_txn_log three-branch routing" into a single helper so that every reader (load_txn_log, publish_log_version, abort_txn, ...) shares one implementation, preventing future drift.
  • Decouple shadow and normal publishes inside publishPartition: a shadow failure should not block the normal side's visibility advancement.
  • Add a FORCE option to CANCEL ALTER TABLE, or allow guarded cancel in FINISHED_REWRITING, so that similar bugs do not permanently lock operators out.
  • Add a metric: starrocks_lake_publish_log_version_failed_total{tablet, reason}, so dashboards can alert directly.
  • Publish a "safe-ALTER matrix": which ALTERs take the FSE v2 fast path vs which require real data backfill. Avoid repeating the Wednesday-call mis-guidance.

Verification (how to confirm the fix took effect)

  • BE log: grep "Fail to publish log version" be.WARNING shows no new entries for 10 minutes.
  • FE log: grep "Fail to publish log version for tablets" fe.warn.log — same.
  • Alter state: in SHOW ALTER TABLE COLUMN, the affected job transitions FINISHED_REWRITING → FINISHED and Progress reaches 100%.
  • Transaction visibility: SHOW TRANSACTION entries previously COMMITTED become VISIBLE; client stream loads succeed within txn_visible_wait_timeout.
  • Timing: expect to see the above within 1–5 minutes after the BE rolling restart completes.

7. What to tell stakeholders

"On May 17 Finaloop's production cluster (StarRocks 4.1.0) experienced a ~5.5-hour write-invisibility incident. The immediate trigger was an ADD COLUMN merchantId ... AS COALESCE(...) — a virtual/generated column ALTER that, by definition, requires a real data backfill. That backfill path collided with a confirmed BE bug (the publish_log_version code path is missing its load_ids branch), which jammed transaction publishing and left the alter job stuck in FINISHED_REWRITING; CANCEL ALTER TABLE is refused by the code in this state.

Responsibility and data loss: we (CelerData) gave Finaloop incorrect prior guidance on a Wednesday call, telling them this ALTER could run safely without a write pause; we did not flag that it would trigger the backfill path. With no hotfix available, Finaloop had to recover via DROP TABLE FORCE + rebuild, which permanently lost the writes that committed but never became visible during the outage.
Fix progress: the BE fix PR is StarRocks/starrocks#73423 (author meegoo, currently Open). It will ship with v4.1.1 and be backported to v4.0.12. JIRA: POST-1518.
Workaround until upgrade: avoid any ALTER that requires data backfill (generated columns, ADD COLUMN with default, key-column changes) during high-traffic windows on PK tables. CelerData will publish a clear safe-ALTER matrix this week.
Next steps: 1) Casey Luo and Luca will sync the root cause with Finaloop; 2) Kevin Chen will publish the safe-ALTER guide by end of week; 3) CelerData will coordinate the v4.1.1 / v4.0.12 upgrade window with Finaloop (or, if any customer needs the patch earlier, a hot-patch BE binary based on PR #73423 can be arranged)."
