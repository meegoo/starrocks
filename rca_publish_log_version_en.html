1. Verdict (one-line)

Not caused by customer workload. This is an internal StarRocks BE bug — publish_log_version() is missing its load_ids branch, so when a table is undergoing a Schema Change while a client commits via SQL BEGIN; INSERT; COMMIT;, BE looks for the source .log at the wrong path, leaving the transaction stuck in COMMITTED forever and the alter job stuck in FINISHED_REWRITING. CANCEL ALTER TABLE is rejected by code in this state, and no non-destructive operator action can break the deadlock.

2. Evidence chain

  1. Cluster baseline: CPU/Mem/Disk/QPS normal, no customer pressure; the publish daemon keeps retrying the same partition.
    — Meaning: this is not a capacity / performance issue, it is a logical deadlock.
  2. Hot objects (concrete IDs):
    • tablet_id = 25883500 (0x18AF36C) and tablet_id = 25883497 (another shadow tablet in the same partition)
    • txn_id = 18149364 (0x114EFF4)
    • label = 019e362a-7257-70d2-b4ff-79dd422b2f8c
    • load_id = 019e362a-725c-7fe1-8d82-07b868ff72b4 (hi=0x019e362a725c7fe1, lo=0x8d8207b868ff72b4)
    • InsertTxnCommitAttachment (rules out HTTP _stream_load; pins to SQL explicit transaction)
    — Meaning: BE's missing filename 00000000018AF36C_000000000114EFF4.log decodes byte-for-byte to (tablet, txn), and the FE label/load_id matches exactly — the evidence is field-aligned.
  3. Trigger cadence: FE PublishVersionDaemon retries the same partition every scheduling tick; BE WARNING fires roughly every second, and never converges (this is not a transient).
    — Meaning: unlike "retried a few times then succeeded" transients, this is a permanent failure loop.
  4. Key logs:
    be.WARNING: 
    Fail to publish log version: Not found:
  starlet err Object s3://.../log/<tablet>_<txn>.log does not exist
  tablet_id=25883500
  txns=[txn_id: 18149364, load_ids { hi:..., lo:... }, ...]
    fe.warn.log (PublishVersionDaemon.publishPartition): 
    Fail to publish log version for tablets [25883500, 25883497]
    Client: Stream load failed: Failed to get load final status, current status: COMMITTED, ...timeout...
    — Meaning: BE reports a path of the form {tablet}_{txn}.log (2-segment), but the actual file on S3 is {tablet}_{txn}_{loadHi}_{loadLo}.log (4-segment).
  5. Resource / thread pool state: BE publish thread pool is not saturated; FE publish daemon has no backlog — it is fast-failing in a loop, so reject policies never fire, and the issue is essentially invisible to resource monitoring.
    — Meaning: this kind of "fast-failing infinite loop" is nearly impossible to spot on dashboards; you have to look at logs or alter job state.
  6. Audit log — shape of the transaction: 
    13:40:11.994  Stmt=BEGIN;                                  QueryId=019e362a-7257-70d2-b4ff-79dd422b2f8c
13:40:11.996  Stmt=insert into businessEventActions select ... left join ...
                                                            QueryId=019e362a-725c-7fe1-8d82-07b868ff72b4
13:40:12.369  Stmt=select last_query_id() ...              (implicit COMMIT already happened)
    — Meaning: the QueryId of BEGIN = txn label, the QueryId of INSERT = load_id, locking the failing operation to "explicit transaction + in-cluster SQL upsert". The customer's HTTP _stream_load pipeline is unrelated.

3. Causal chain (internal anomaly → customer-visible symptom)

  1. Precondition: the customer runs ADD COLUMN ... AS COALESCE(...) on businessEventActions (backfilling Schema Change). The shadow index (IndexState.SHADOW) is created and the alter enters the backfill phase.
  2. Transaction commit: the customer commits a SQL BEGIN; INSERT INTO businessEventActions SELECT ... ; COMMIT; during the alter window. FE writes TransactionState.loadIds into TxnInfoPB.load_ids.
  3. BE write: delta_writer calls TabletManager::put_txn_log; because has_load_id() is true, the .log lands at the 4-segment path with the load_id suffix (tablet_manager.cpp:907).
  4. FE publish: PublishVersionDaemon.publishPartition splits tablets by MaterializedIndex.IndexState — normal tablets go through publishVersion, shadow tablets through publishLogVersion (PublishVersionDaemon.java:1062-1083).
  5. BE read bug: publish_log_version() only branches on combined_txn_log; there is no load_ids_size() > 0 branch. It calls fs::copy_file() on the 2-segment path → NotFound (transactions.cpp:507-542). The idempotency fallback requires the target .vlog to already exist, but it never has.
  6. Exception swallowing: publishPartition wraps publishLogVersion and publishVersion in the same try block; when the former throws, the latter never runs, and the catch returns false (PublishVersionDaemon.java:1081-1110).
  7. Transaction frozen: the partition's publish failed → the transaction's commitVersion never becomes visibleVersion → the transaction remains in COMMITTED forever.
  8. Alter deadlock: LakeTableSchemaChangeJob.readyToPublishVersion() requires commitVersion == visibleVersion + 1 to advance into FINISHED; the condition is never satisfied (LakeTableSchemaChangeJob.java:805-820) → alter loops in FINISHED_REWRITING forever.
  9. Recovery blocked: an operator running CANCEL ALTER TABLE hits cancelImpl's guard at line 1166 — jobState == FINISHED_REWRITING && tableExists() → returns false immediately, and the cancel is silently rejected.
  10. Customer-visible symptoms:
    • Explicit-transaction clients hit txn_visible_wait_timeout and return errors;
    • All new writes to that table become invisible (subsequent stream loads / INSERTs on the same partition also traverse the shadow path and stall publish);
    • SHOW ALTER TABLE COLUMN shows the job permanently stuck in FINISHED_REWRITING;
    • BE/FE logs keep printing the same warnings; dashboards show no resource anomaly.

4. Contributing factors

  • Read/write routing asymmetry: the writer put_txn_log routes on has_load_id() (2 branches); the reader load_txn_log() has all three branches; but publish_log_version() only has two — the same routing protocol is reimplemented in three places and one of them drifted.
  • Exception bundling: publishPartition wraps shadow and normal publishes in the same try block; an exception from the shadow side blocks the normal side from advancing visibility, turning a localized failure into a whole-partition outage.
  • Cancel-during-FINISHED_REWRITING is forbidden by design: this guard protects data integrity (the alter is near completion at this stage and a forced cancel could lose data), but in this bug it locks operators out, and there is no FORCE / break-glass bypass.
  • Idempotency fallback is unusable in "never succeeded" cases: the existing fallback only short-circuits when the .vlog already exists. It cannot recognize a persistent "source path is wrong" failure.
  • Mixed customer pipeline: the customer also runs HTTP _stream_load staging, so on first inspection the failure looks like a stream-load issue. You have to inspect InsertTxnCommitAttachment (vs StreamLoadTxnCommitAttachment) and the origin of loadIds (only TransactionStmtExecutor.commitStmt writes it) to disambiguate.

5. Why X failed — why the usual safety nets did not catch this

  • FE retries: FE classifies NotFound as retryable, but every retry hits the same dead path, with no upper bound on retries.
  • Idempotency probe: on NotFound, publish_log_version probes whether the target .vlog already exists; but the .vlog was never generated (the 2-segment source .log never existed), the probe always misses, and the fallback degenerates to returning the original error.
  • Alter watershedTxnId barrier: the alter's WAITING_TXN phase waits for all txns with txn_id < watershedTxnId to finish, but this bug's triggering transactions are committed after the alter started (txn_id > watershedTxnId), so the barrier doesn't protect them.
  • CANCEL ALTER TABLE: the default logic refuses to cancel in FINISHED_REWRITING (assuming the job is already nearly complete and shouldn't be rolled back); there is no FORCE override.
  • Monitoring: CPU/Mem/QPS dashboards look entirely healthy; publish_log_version's WARNING has no dedicated metric counter; you have to inspect alter job state to notice anything is wrong.

6. Recommended actions (in priority order)

Step 1 — Fix the root cause: BE hotfix for publish_log_version()

Add a txn_info.load_ids_size() > 0 branch to publish_log_version() in be/src/storage/lake/transactions.cpp:507, mirroring the three-branch structure of load_txn_log() (same file, line 123): read each per-load_id .log from the 4-segment path, merge them into a single TxnLog, write one .vlog, queue source files for async deletion.

// Fix sketch (as the first branch of the for-loop)
if (txn_infos[i].load_ids_size() > 0) {
    ASSIGN_OR_RETURN(auto per_tablet_logs,
                     load_txn_log(tablet_mgr, {tablet_id}, txn_infos[i]));
    auto& logs = per_tablet_logs[0];
    if (logs.empty()) {
        auto vlog_path = tablet_mgr->txn_vlog_location(tablet_id, log_versions[i]);
        ASSIGN_OR_RETURN(auto fs, FileSystemFactory::CreateSharedFromString(vlog_path));
        if (fs->path_exists(vlog_path).ok()) continue;  // idempotent
        return Status::NotFound(...);
    }
    auto merged = merge_txn_logs(logs);
    RETURN_IF_ERROR(tablet_mgr->put_txn_vlog(merged, log_versions[i]));
    for (const auto& load_id : txn_infos[i].load_ids()) {
        files_to_delete.emplace_back(
            tablet_mgr->txn_log_location(tablet_id, txn_infos[i].txn_id(), load_id));
    }
    continue;
}

Expected effect: once the BE binary is replaced and rolling-restarted, the next publish scheduler tick (~1s by default) will successfully produce the .vlog and the partition's visibleVersion will advance to commitVersion; on the following LakeTableSchemaChangeJob tick, the alter transitions to FINISHED and the shadow index is promoted. No data loss.

Step 2 — Stop the bleeding (before the hotfix ships)

  1. Identify affected alter jobs: SHOW ALTER TABLE COLUMN FROM <db>; look for jobs with State = FINISHED_REWRITING and no progress for a long time, on tables that also receive SQL explicit-transaction writes.
  2. For affected tables, temporarily switch the application off the "explicit transaction + INSERT" path (use HTTP _stream_load or implicit-transaction INSERT) — this does not recover the already-stuck alter, but it prevents new transactions from piling onto the deadlock queue.
  3. If business cannot wait for the hotfix and accepts data loss: DROP TABLE <name> so that tableExists() returns false and cancelImpl can proceed past the guard. Cost: the entire table's data is lost and must be rebuilt from upstream.

Do NOT: keep retrying CANCEL ALTER TABLE, restart FE/BE in hopes of recovery, or manually delete .log files from S3 — none of these unblock the cancel, and the last option may cause real data loss.

Step 3 — Long-term hardening

  • Extract the "has_load_id() / combined_txn_log three-branch routing" into a single helper so that every reader (load_txn_log, publish_log_version, abort_txn, ...) shares one implementation, preventing future drift.
  • Decouple shadow and normal publishes inside publishPartition: a shadow failure should not block the normal side's visibility advancement (at minimum: when NotFound happens but the 4-segment path file exists, do not bubble the exception).
  • Add a FORCE option to CANCEL ALTER TABLE, or allow guarded cancel in FINISHED_REWRITING, so that similar bugs do not permanently lock operators out.
  • Add a metric: starrocks_lake_publish_log_version_failed_total{tablet, reason}, so dashboards can alert directly.

Verification (how to confirm the fix took effect)

  • BE log: grep "Fail to publish log version" be.WARNING shows no new entries; existing entries' timestamps are frozen; nothing new appears for 10 minutes.
  • FE log: grep "Fail to publish log version for tablets" fe.warn.log — same.
  • Alter state: in SHOW ALTER TABLE COLUMN, the affected job transitions FINISHED_REWRITING → FINISHED and Progress reaches 100%.
  • Transaction visibility: SHOW TRANSACTION entries previously COMMITTED become VISIBLE; insert_visible_latency returns to baseline.
  • Customer validation: a client re-issuing BEGIN; INSERT; COMMIT; returns success within txn_visible_wait_timeout.
  • Timing: expect to see the above within 1–5 minutes after the BE rolling restart completes (driven by the publish daemon and alter scheduler ticks).

7. What to tell stakeholders

"We've identified an internal bug in the StarRocks BE under shared-data mode: when a table is undergoing a Schema Change (e.g. ADD COLUMN ... AS COALESCE(...)) and a client commits data through a SQL BEGIN; INSERT; COMMIT; explicit transaction at the same time, BE looks up the source log at the wrong path and cannot publish that version. The Schema Change job then stalls at the finishing step, and newly written rows remain temporarily invisible to queries.

No data loss: every file the client wrote is intact on shared storage.
Recovery path: upgrade to a BE build containing the fix (see PR #xxxx, planned for v3.5.x). After the upgrade, queued transactions catch up within minutes and the Schema Change finishes on its own.
Temporary workaround: until the upgrade is in place, please avoid SQL BEGIN; INSERT; COMMIT; on tables that have a Schema Change in flight. HTTP _stream_load and implicit-transaction INSERT are not affected.
Customer actions: 1) wait for the BE hotfix; 2) pause explicit-transaction writes on the affected tables; 3) coordinate an upgrade window if downtime sensitivity is high."
