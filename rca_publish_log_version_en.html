RCA: publish_log_version Ignores load_ids — Explicit Transactions Stuck COMMITTED During Schema Change

ClassificationBugFix / Shared-data Lake / Transaction Visibility
Affected modulesbe/src/storage/lake/transactions.cpp (publish_log_version)
Trigger(1) Table has a backfilling schema change in progress; (2) Transaction is committed via SQL BEGIN; INSERT ...; COMMIT;
ImpactExplicit-transaction INSERTs remain COMMITTED until the alter finishes; clients time out on txn_visible_wait_timeout
Data safetyNo data loss; the 4-segment .log files exist on storage and visibility catches up after the alter completes

1. Summary

During a schema change, FE calls publishLogVersionBatch so BE can materialize each transaction's .log on shadow-index tablets into a versioned .vlog. The BE function publish_log_version() always builds the 2-segment path (no load_id suffix) to locate the source .log. For transactions committed via SQL BEGIN; INSERT ...; COMMIT;, the writer places the log at the 4-segment path (with load_id suffix). BE returns NotFound, FE retries forever, and the transaction stays COMMITTED until the schema change finishes.

for (int i = 0; i < txn_infos.size(); i++) {

        if (st.is_not_found()) {
            // idempotency fallback: check whether the target .vlog already exists
            ...
            return st;                                                 // ❌ real failure returned to FE
        }
    } else {
        ASSIGN_OR_RETURN(auto txn_logs, load_txn_log(tablet_mgr, {tablet_id}, txn_infos[i]));

The txn_info.load_ids_size() > 0 branch is missing. When the request carries load_ids, BE reads the 2-segment path and always returns NotFound.

5. Why "Normal" Loads Are Unaffected

Answer: the bug is always present, but the buggy code path is normally never reached. FE in PublishVersionDaemon.publishPartition() splits tablets by their MaterializedIndex.IndexState: 
// fe/.../PublishVersionDaemon.java:1062-1074
for (MaterializedIndex index : indexes) {
    if (index.getState() == MaterializedIndex.IndexState.SHADOW) {
        shadowTablets.addAll(index.getTablets());   // → publishLogVersion (buggy path)
    } else {
        normalTablets.addAll(index.getTablets());   // → publishVersion (correct path)
    }
}
FE methodBE RPCBE core functionload_ids branch
Utils.publishVersionpublish_versionpublish_version() → load_txn_log()✅ Yes
Utils.publishLogVersion(Batch)publish_log_version(_batch)publish_log_version()❌ No

IndexState.SHADOW exists only transiently during a schema change. Once the alter completes, the shadow index is promoted to NORMAL and subsequent publishes go through publish_version() — the correct path. Therefore:

  • No active alter: shadowTablets is always empty, the buggy path is unreachable, everything works.
  • Active alter: explicit-transaction INSERTs are stuck COMMITTED with 100% reproducibility until the alter completes.
  • After the alter: the backlog catches up automatically, no data loss.

6. Who Populates load_ids

TxnInfoPB.load_ids has exactly one writer in FE — TxnInfoHelper.java:41, gated by a strict condition:

if (state.getLoadIds() != null
        && state.getSourceType() == TransactionState.LoadJobSourceType.INSERT_STREAMING) {
    infoPB.setLoadIds(...);
}

TransactionState.loadIds is set exclusively by TransactionStmtExecutor.commitStmt. Conclusion: load_ids is non-empty iff the transaction was committed through SQL BEGIN; ... COMMIT;. HTTP _stream_load and HTTP multi-statement transaction load do not set the field and do not trigger the bug.

8. Why Retries Cannot Recover

The idempotency fallback depends on the target .vlog already existing; however, the source 2-segment .log never existed, so the .vlog is never written. Every retry returns NotFound. The only exits are:

  1. The schema change completes — the shadow index is promoted to NORMAL, and subsequent publishes go through publish_version(), which correctly reads the 4-segment .log.
  2. The alter is cancelled.

9. Impact

  • Every explicit-transaction INSERT that runs during a backfilling alter is stuck at COMMITTED until the alter finishes.
  • Clients with txn_visible_wait_timeout shorter than the alter duration time out.
  • The publish daemon backlog can stall unrelated stream loads on the same node.
  • No data loss: the source .log files exist; visibility catches up after the alter completes.

11. Fix Direction

Add a txn_info.load_ids_size() > 0 branch to publish_log_version(), mirroring load_txn_log().

11.1 Fix sketch for publish_log_version

if (txn_infos[i].load_ids_size() > 0) {
    ASSIGN_OR_RETURN(auto per_tablet_logs,
                     load_txn_log(tablet_mgr, {tablet_id}, txn_infos[i]));
    auto& logs = per_tablet_logs[0];

    // Idempotency fallback: skip if the target .vlog already exists.
    if (logs.empty()) {
        auto vlog_path = tablet_mgr->txn_vlog_location(tablet_id, log_versions[i]);
        ASSIGN_OR_RETURN(auto fs, FileSystemFactory::CreateSharedFromString(vlog_path));
        if (fs->path_exists(vlog_path).ok()) continue;
        return Status::NotFound(...);
    }

    auto merged = merge_txn_logs(logs);                     // reuse existing applier merge logic
    RETURN_IF_ERROR(tablet_mgr->put_txn_vlog(merged, log_versions[i]));
    for (const auto& load_id : txn_infos[i].load_ids()) {
        files_to_delete.emplace_back(
            tablet_mgr->txn_log_location(tablet_id, txn_infos[i].txn_id(), load_id));
    }
    continue;
}

11.2 Test recommendations

  • BE UT: construct TxnInfoPB.load_ids with multiple load_ids, pre-write 4-segment .log files, assert that publish_log_version produces a single correct .vlog and queues the source files for deletion. Cover the idempotency case (target .vlog already exists).
  • Regression integration: explicit-transaction INSERT concurrent with ADD COLUMN ... AS COALESCE(...); assert visibility within txn_visible_wait_timeout and that COMMITTED → VISIBLE progresses normally.

12. Risks

  • Merge semantics: merging multiple per-load_id TxnLogs must match the applier used by publish_version() to avoid divergence between the shadow and normal indexes. Reuse the existing applier path rather than rewriting it.
  • Cleanup timing: retry must be able to re-read the source file; keep the existing delete_files_async pattern instead of synchronous deletion in the main path.
  • Code duplication: the combined_txn_log branch already delegates to load_txn_log; the new load_ids branch should do the same to prevent future drift.
