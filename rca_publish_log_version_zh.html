RCA：publish_log_version 忽略 load_ids 导致 Schema Change 死锁

问题分类BugFix / Shared-data Lake / 事务可见性 / Schema Change 死锁
影响模块be/src/storage/lake/transactions.cpp（publish_log_version）
触发条件① 表正在执行 Schema Change（影子索引存在）；② 同时有事务通过 SQL BEGIN; INSERT ...; COMMIT; 提交
影响范围显式事务 INSERT 永远卡在 COMMITTED；Schema Change 进入 FINISHED_REWRITING 后永远无法完成；阻塞同分区后续所有事务的可见
解除方式仅能通过 CANCEL ALTER TABLE 取消 Schema Change
数据安全无数据丢失（4 段 .log 实际存在于共享存储），但分区可见版本被永久冻结，业务持续不可见新数据

1. 问题摘要

Schema change 期间，FE 通过 publishLogVersionBatch 让 BE 把影子（shadow）索引 tablet 上的事务日志 .log 物化为版本化的 .vlog。BE 函数 publish_log_version() 总是按 2 段路径（不带 load_id 后缀）去找源 .log。但通过 SQL BEGIN; INSERT ...; COMMIT; 提交的事务，写端会把日志写到 4 段路径（带 load_id 后缀）。BE 永远 NotFound，publishPartition 抛异常返回 false，事务永远停留在 COMMITTED。

更严重的是，这条事务永远无法 VISIBLE，导致分区的 visibleVersion 被永久冻结，而 Schema Change 在 FINISHED_REWRITING 阶段必须等待 visibleVersion == commitVersion - 1，因此 alter 永远无法收尾——形成"事务等 alter、alter 等事务"的死锁，唯一解除方法是 CANCEL ALTER TABLE。

5. 为什么"正常导入"没事？

答：bug 一直存在，但平时不会被触达。FE 在 PublishVersionDaemon.publishPartition() 中按 MaterializedIndex.IndexState 将 tablet 分流： 
// fe/.../PublishVersionDaemon.java:1062-1074
for (MaterializedIndex index : indexes) {
    if (index.getState() == MaterializedIndex.IndexState.SHADOW) {
        shadowTablets.addAll(index.getTablets());   // → publishLogVersion（缺陷路径）
    } else {
        normalTablets.addAll(index.getTablets());   // → publishVersion（正确路径）
    }
}
FE 方法BE RPCBE 核心函数load_ids 分支
Utils.publishVersionpublish_versionpublish_version() → load_txn_log()✅ 有
Utils.publishLogVersion(Batch)publish_log_version(_batch)publish_log_version()❌ 没有

IndexState.SHADOW 仅在 Schema Change 执行期间存在。没有 alter 时 shadowTablets 永远为空，bug 路径不会被触达，所以"正常导入"全部走 normal 路径，命中 load_txn_log 的 load_ids 分支，能正确读到 4 段 .log。

6. 谁会让 load_ids 非空

TxnInfoPB.load_ids 仅由 fe/.../TxnInfoHelper.java:41 写入，且条件严格：

if (state.getLoadIds() != null
        && state.getSourceType() == TransactionState.LoadJobSourceType.INSERT_STREAMING) {
    infoPB.setLoadIds(...);
}

TransactionState.loadIds 仅由 TransactionStmtExecutor.commitStmt 写入。结论：load_ids 非空 ⇔ SQL 显式事务 BEGIN; ... COMMIT;。HTTP _stream_load、HTTP 多语句事务 load 均不会触发本 bug。

8. Schema Change 死锁机制（关键）

这不是"等 alter 跑完就好了"——alter 本身永远不会跑完，需要人工介入。
任意一个 显式事务（携带 load_ids）在影子索引存在期间提交，就会让 alter 永远卡在 FINISHED_REWRITING 阶段。

8.1 致死环

  1. Schema Change 创建影子索引（IndexState.SHADOW），进入回填阶段；
  2. 客户端在 alter 进行中发起 BEGIN; INSERT INTO ... ; COMMIT;，该事务 TxnInfoPB.load_ids 非空；
  3. 事务的 .log 被 BE 按 4 段路径写入（含 normal tablet 与 shadow tablet）；
  4. FE PublishVersionDaemon 调用 publishPartition，对该分区同时执行：
    • Utils.publishLogVersion(shadowTablets, ...) → BE publish_log_version() → 按 2 段路径找不到 → NotFound
    • Utils.publishVersion(normalTablets, ...) → 本来能成功，但被前一步的异常打断，不会执行
  5. publishPartition 进入 catch 分支，setErrorMsg 后返回 false（PublishVersionDaemon.java:1099-1110）；
  6. 整个分区当前 commit 没有 publish 成功 → 事务在该分区上的 commitVersion 没有变成 visibleVersion；
  7. 事务状态永远停在 COMMITTED；
  8. Schema Change 在 FINISHED_REWRITING 阶段调用 readyToPublishVersion()， 在 LakeTableSchemaChangeJob.java:813 校验 commitVersion == physicalPartition.getVisibleVersion() + 1； 因为 visibleVersion 不再前进，该条件永远不成立 → alter 在此循环中无限等待；
  9. 影子索引保持 SHADOW 状态 → publishLogVersion 继续被调用 → 继续失败 → 回到第 4 步形成无限循环。

8.3 死锁示意

┌──────────────────────────────────────────────────────────────────┐ │ 显式事务 T │ │ status = COMMITTED, visibleVersion 不前进 │ └──────────────────────────────────────────────────────────────────┘ ↑ │ │ publish 永远失败 │ 阻止 visibleVersion 推进 │ ↓ ┌──────────────────────────────────────────────────────────────────┐ │ publish_log_version() (BE) │ │ 按 2 段路径找不到 → NotFound │ └──────────────────────────────────────────────────────────────────┘ ↑ │ │ FE 持续调度重试 │ partition 卡在 COMMITTED │ ↓ ┌──────────────────────────────────────────────────────────────────┐ │ Schema Change 阶段 = FINISHED_REWRITING │ │ readyToPublishVersion() 等 visibleVersion+1 == commitVersion │ │ → 条件永远不成立 → 影子索引永远保持 SHADOW 状态 │ └──────────────────────────────────────────────────────────────────┘

9. 唯一退出路径

没有任何自动恢复机制。幂等回退依赖目标 .vlog 已存在；而 .vlog 永远不会被生成（源文件按 2 段路径根本不存在）。重试在死锁解除前都会 NotFound。

唯一让系统脱困的操作：

CANCEL ALTER TABLE <table_name> COLUMN;

cancel 之后：

  • 影子索引被丢弃，shadowTablets 集合变空；
  • publishPartition 不再调用 publishLogVersion，publishVersion（normal tablets）单独执行成功；
  • 积压的 COMMITTED 事务在下一轮 publish 守护进程调度中转为 VISIBLE；
  • 用户需要在业务允许的窗口期重新发起 Schema Change。

10. 影响

  • 触发后 Schema Change 永远停在 FINISHED_REWRITING，alter job 不会自然结束。
  • 所有显式事务 INSERT 在 alter 期间永远 COMMITTED。
  • 同一分区上后续 普通 stream load / INSERT 也无法可见（因为它们的 publishPartition 同样会经过 shadow tablet 路径，同样会抛异常）。
  • 客户端 txn_visible_wait_timeout 全部超时返回。
  • publish 守护进程持续重试，堆积线程占用与日志噪声，可能拖慢同节点上的其他分区 publish。
  • 无数据丢失：源 .log 与 segment 完整保留在共享存储；cancel 后通过 normal 路径补齐可见性。
