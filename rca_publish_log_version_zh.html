1. Verdict（一句话定性）

不是用户工作负载导致；是 StarRocks BE 的内部 bug —— publish_log_version() 缺少 load_ids 分支，在表正进行 Schema Change 且同时有 SQL BEGIN; INSERT; COMMIT; 提交时，BE 用错误的 .log 路径查找，让事务永远停在 COMMITTED 状态，并把 alter job 卡死在 FINISHED_REWRITING 阶段。CANCEL ALTER TABLE 在该状态下被代码强制拒绝，运维侧无非破坏性恢复手段。

2. Evidence chain（证据链）

  1. 集群基线：CPU/Mem/Disk/QPS 正常，无用户压力；publish 守护线程持续重试同一个分区。
    — 说明：故障不是性能瓶颈或容量问题，是逻辑死锁。
  2. 热点对象（具体 ID）：
    • tablet_id = 25883500（0x18AF36C）、tablet_id = 25883497（同一分区另一影子 tablet）
    • txn_id = 18149364（0x114EFF4）
    • label = 019e362a-7257-70d2-b4ff-79dd422b2f8c
    • load_id = 019e362a-725c-7fe1-8d82-07b868ff72b4（hi=0x019e362a725c7fe1, lo=0x8d8207b868ff72b4）
    • InsertTxnCommitAttachment（排除 HTTP _stream_load，定位为 SQL 显式事务）
    — 说明：BE 报错文件名 00000000018AF36C_000000000114EFF4.log 解码恰好等于 (tablet, txn)，与 FE 日志的 label/load_id 精确对应，可逐字段比对。
  3. 触发频率：FE PublishVersionDaemon 默认调度周期内每轮都对该分区重试，BE WARNING 每秒级别刷出，且永不收敛（不是间歇性问题）。
    — 说明：与典型的"重试若干次后成功"的暂态错误不同，这是一条永久失败链路。
  4. 关键日志：
    be.WARNING： 
    Fail to publish log version: Not found:
  starlet err Object s3://.../log/<tablet>_<txn>.log does not exist
  tablet_id=25883500
  txns=[txn_id: 18149364, load_ids { hi:..., lo:... }, ...]
    fe.warn.log（PublishVersionDaemon.publishPartition）： 
    Fail to publish log version for tablets [25883500, 25883497]
    客户端：Stream load failed: Failed to get load final status, current status: COMMITTED, ...timeout...
    — 说明：BE 报的路径形如 {tablet}_{txn}.log（2 段路径），但 S3 上实际存在的是 {tablet}_{txn}_{loadHi}_{loadLo}.log（4 段路径）。
  5. 资源 / 线程池状态：BE 的 publish 线程池没饱和，FE publish daemon 没堆积——它持续在快速失败，所以反而不会触发 reject 策略，问题靠资源监控看不出来。
    — 说明：这种"快速失败的死循环"在 dashboard 上几乎不可见，必须看日志或 alter job 状态才能发现。
  6. 审计日志中的事务形态： 
    13:40:11.994  Stmt=BEGIN;                                  QueryId=019e362a-7257-70d2-b4ff-79dd422b2f8c
13:40:11.996  Stmt=insert into businessEventActions select ... left join ...
                                                            QueryId=019e362a-725c-7fe1-8d82-07b868ff72b4
13:40:12.369  Stmt=select last_query_id() ...              (隐式 COMMIT 已发生)
    — 说明：QueryId of BEGIN = txn label，QueryId of INSERT = load_id，精确锁定到"显式事务 + 内部 SQL upsert"这一形态。客户的 HTTP _stream_load 流量与此 bug 无关。

3. Causal chain（因果链 → 客户可见症状）

  1. 触发前提：客户对 businessEventActions 表执行 ADD COLUMN ... AS COALESCE(...)（回填型 Schema Change），影子索引（IndexState.SHADOW）建好，alter 进入回填阶段。
  2. 事务提交：客户在 alter 进行期间通过 SQL BEGIN; INSERT INTO businessEventActions SELECT ... ; COMMIT; 提交事务，FE 把 TransactionState.loadIds 写入 TxnInfoPB.load_ids。
  3. BE 写入：delta_writer 调用 TabletManager::put_txn_log，因为 has_load_id() 为 true，.log 落到带 load_id 后缀的 4 段路径（tablet_manager.cpp:907）。
  4. FE 发布：PublishVersionDaemon.publishPartition 按 MaterializedIndex.IndexState 分流——normal tablets 走 publishVersion，shadow tablets 走 publishLogVersion（PublishVersionDaemon.java:1062-1083）。
  5. BE 读端 bug：BE publish_log_version() 只判 combined_txn_log，没有 load_ids_size() > 0 分支，按 2 段路径 fs::copy_file() → NotFound（transactions.cpp:507-542）。幂等回退要求目标 .vlog 已存在，但它从未被生成。
  6. 异常吞并：publishPartition 的 try 块把 publishLogVersion 和 publishVersion 串行包在一起；前者抛出后，后者根本不会被执行，整个 catch 返回 false（PublishVersionDaemon.java:1081-1110）。
  7. 事务冻结：该 partition 的 publish 失败 → txn 的 commitVersion 永远不变 visibleVersion → 事务永远 COMMITTED。
  8. Alter 死锁：LakeTableSchemaChangeJob.readyToPublishVersion() 要求 commitVersion == visibleVersion + 1 才能进入 FINISHED；条件永远不成立（LakeTableSchemaChangeJob.java:805-820）→ alter 在 FINISHED_REWRITING 状态无限等待。
  9. 恢复被堵死：用户尝试 CANCEL ALTER TABLE，cancelImpl 在 1166 行判断 jobState == FINISHED_REWRITING && tableExists() → 直接 return false，cancel 被静默拒绝。
  10. 客户可见症状：
    • 显式事务客户端在 txn_visible_wait_timeout 后超时；
    • 该表新写入数据全部不可见（同分区后续 stream load/INSERT 同样会经过 shadow 路径，同样卡 publish）；
    • SHOW ALTER TABLE COLUMN 显示 job 永远停在 FINISHED_REWRITING；
    • BE/FE 日志反复刷出固定 warning，dashboard 无明显资源异常。

4. Contributing factors（放大因素 / 易混淆项）

  • 读写端路由不对称：写端 put_txn_log 按 has_load_id() 分两路；读端 load_txn_log() 三分支齐全；但 publish_log_version() 只两分支——同一份"路由协议"在三个函数里各写一遍，导致漂移。
  • 异常打包：publishPartition 把 shadow + normal 两个 publish 顺序放在同一 try 块，shadow 抛异常时连带阻塞 normal 的可见性推进，把局部故障放大成全分区故障。
  • FINISHED_REWRITING 禁止 cancel 的设计：原本是为了保护数据完整性（这阶段已经把 shadow 推到接近完成，强行 cancel 可能丢数据），但在本 bug 场景下反而堵死了运维通道，且没有force/break-glass 旁路。
  • 幂等回退在"从未成功"场景下不可用：现有回退仅在之前成功过（.vlog 已存在）时跳过，无法识别"源文件路径就错"这一类持久性错误。
  • 客户管线混合：客户同时有 HTTP _stream_load staging + SQL 显式事务 upsert，初期容易被误判为 stream load 问题；要靠 InsertTxnCommitAttachment 类型字段 + loadIds 来源（仅 TransactionStmtExecutor.commitStmt 写入）才能区分。

5. Why X failed —— 为什么常规护栏没生效

  • FE 重试机制：FE 把 NotFound 当作可重试错误，但每次重试都打到同一条死路径，重试次数无上限。
  • 幂等回退（idempotency probe）：publish_log_version 在 NotFound 时会探测 .vlog 是否已存在；但.vlog 从未被生成（源 2 段 .log 根本不存在），探测必然失败，回退退化为直接 return error。
  • Alter 的 watershedTxnId 屏障：alter 在 WAITING_TXN 阶段会等所有 txn_id < watershedTxnId 的事务完成，但本 bug 触发的事务是 alter 启动之后才提交的（txn_id > watershedTxnId），不在屏障保护范围内。
  • CANCEL ALTER TABLE 命令：默认逻辑在 FINISHED_REWRITING 阶段拒绝 cancel（认为该阶段已经几乎完成、不该回退），没有 FORCE 选项。
  • 监控告警：dashboard 看 CPU/Mem/QPS 一切正常，publish_log_version 的 WARNING 没有专门的 metric 计数，需要看 alter job 状态才能感知。

6. Recommended actions（按优先级排序）

Step 1 — 修根因：BE 热修复 publish_log_version()

在 be/src/storage/lake/transactions.cpp:507 的 publish_log_version() 中加入 txn_info.load_ids_size() > 0 分支，与 load_txn_log()（同文件 line 123）的三分支模式对齐：按 4 段路径读取每个 load_id 的 .log，合并写出一个 .vlog，源文件入 files_to_delete 异步清理。

// 修复骨架（在原 for 循环中作为第一个分支）
if (txn_infos[i].load_ids_size() > 0) {
    ASSIGN_OR_RETURN(auto per_tablet_logs,
                     load_txn_log(tablet_mgr, {tablet_id}, txn_infos[i]));
    auto& logs = per_tablet_logs[0];
    if (logs.empty()) {
        auto vlog_path = tablet_mgr->txn_vlog_location(tablet_id, log_versions[i]);
        ASSIGN_OR_RETURN(auto fs, FileSystemFactory::CreateSharedFromString(vlog_path));
        if (fs->path_exists(vlog_path).ok()) continue;  // 幂等
        return Status::NotFound(...);
    }
    auto merged = merge_txn_logs(logs);
    RETURN_IF_ERROR(tablet_mgr->put_txn_vlog(merged, log_versions[i]));
    for (const auto& load_id : txn_infos[i].load_ids()) {
        files_to_delete.emplace_back(
            tablet_mgr->txn_log_location(tablet_id, txn_infos[i].txn_id(), load_id));
    }
    continue;
}

预期效果：BE 二进制替换 + 滚动重启完成后，下一次 publish 调度（默认 1s 周期）就会成功生成 .vlog，分区 visibleVersion 跳到 commitVersion；LakeTableSchemaChangeJob 在再下一轮调度中进入 FINISHED，影子索引升正、alter 自然结束。对存量数据无损。

Step 2 — 应急止血（在热修复发布之前）

  1. 识别中招的 alter job：SHOW ALTER TABLE COLUMN FROM <db>，找 State = FINISHED_REWRITING 且长时间无进展、且对应表有 SQL 显式事务写入的 job。
  2. 对受影响表，临时禁用业务侧的"显式事务 + INSERT"路径（改走 HTTP _stream_load 或者隐式事务 INSERT），避免新事务进一步加入死锁队列。注意：这不会救回已经卡住的 alter，只能止血。
  3. 若业务无法等待热修复发布，且接受数据丢失：执行 DROP TABLE <name> 让 tableExists() 返回 false，cancelImpl 才能继续走完取消流程。代价：整张表数据丢失，必须能从上游重建。

不推荐操作：连续重试 CANCEL ALTER TABLE、重启 FE/BE、强删 S3 上的 .log 文件——这些都不会让 cancel 生效，且第三项可能造成真正的数据丢失。

Step 3 — 长期改进

  • 把"has_load_id() / combined_txn_log 三分支路由"抽成单一帮助函数，让所有读端（load_txn_log、publish_log_version、abort_txn 等）共享，防止下次漂移。
  • 在 publishPartition 中把 shadow 与 normal 两段 publish 解耦：shadow 失败不应阻塞 normal 的可见性推进（至少做到 NotFound + 4 段路径已存在时不抛异常）。
  • 给 CANCEL ALTER TABLE 增加 FORCE 选项或在 FINISHED_REWRITING 状态下允许带保护条件的 cancel，避免类似 bug 把运维通道彻底堵死。
  • 新增 metric：starrocks_lake_publish_log_version_failed_total{tablet, reason}，让 dashboard 能直接报警。

Verification（怎么确认修好了）

  • BE 日志：grep "Fail to publish log version" be.WARNING 不再有新输出（旧条目时间戳冻结），10 分钟内无新增。
  • FE 日志：grep "Fail to publish log version for tablets" fe.warn.log 同上。
  • Alter 状态：SHOW ALTER TABLE COLUMN 中受影响 job 的 State 从 FINISHED_REWRITING → FINISHED，Progress 推到 100%。
  • 事务可见性：SHOW TRANSACTION 列出的之前 COMMITTED 事务转为 VISIBLE，QPS 维度上 insert_visible_latency 回落到基线。
  • 客户验证：客户端重新执行 BEGIN; INSERT; COMMIT; 在 txn_visible_wait_timeout 内成功返回。
  • 时间预期：BE 滚动重启完成后 1–5 分钟内可见上述变化（取决于 publish daemon 周期与 alter scheduler tick）。

7. What to tell stakeholders（对外口径）

"我们在 StarRocks 共享存储模式下发现一个内部 bug：当一张表正在进行 Schema Change（例如 ADD COLUMN ... AS COALESCE(...)），并且同时有客户端通过 SQL BEGIN; INSERT; COMMIT; 显式事务写入数据时，BE 会因路径查找错误而无法发布版本，导致 Schema Change 卡在收尾阶段无法结束，新写入数据暂时不可见。

无数据丢失：所有写入的文件都完整保留在共享存储上。
恢复路径：需要升级到包含修复的 BE 版本（patch 见 PR #xxxx，预计在 v3.5.x 发布）。升级后积压事务会在分钟级自动追平、Schema Change 自动收尾。
临时规避：在升级完成前，请避免在 Schema Change 进行期间对该表执行 SQL 显式事务 INSERT；HTTP _stream_load 和隐式事务 INSERT 不受影响。
客户行动：1) 等待我们提供的 BE 热补丁；2) 暂停影响表上的显式事务写入；3) 如对停机敏感，请协调升级窗口。"
