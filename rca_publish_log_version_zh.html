RCA：publish_log_version 忽略 load_ids 导致显式事务在 Schema Change 期间卡 COMMITTED

问题分类BugFix / Shared-data Lake / 事务可见性
影响模块be/src/storage/lake/transactions.cpp（publish_log_version）
触发条件① 表正在执行回填型 Schema Change；② 事务通过 SQL BEGIN; INSERT ...; COMMIT; 提交
影响范围显式事务 INSERT 在 alter 完成前持续 COMMITTED；客户端在 txn_visible_wait_timeout 内超时
数据安全无数据丢失；4 段 .log 实际存在，alter 结束后可见性自动追平

1. 问题摘要

Schema change 期间，FE 通过 publishLogVersionBatch 让 BE 把影子（shadow）索引 tablet 上的事务日志 .log 物化为版本化的 .vlog。BE 函数 publish_log_version() 总是按 2 段路径（不带 load_id 后缀）去找源 .log。但通过 SQL BEGIN; INSERT ...; COMMIT; 提交的事务，写端会把日志写到 4 段路径（带 load_id 后缀）。BE 找不到文件返回 NotFound，FE 无限重试。事务持续 COMMITTED，直到 alter 结束。

2. 现象

2.1 BE 日志（反复输出同一路径）

Fail to publish log version: Not found:
  starlet err Object s3://.../log/<tablet>_<txn>.log does not exist
  tablet_id=25883500
  txns=[txn_id: 18149364, load_ids { hi:..., lo:... }, ...]

2.2 FE 日志

PublishVersionDaemon.publishPartition 持续输出： Fail to publish log version for tablets [25883500, 25883497]

2.3 客户端

Stream load failed: Failed to get load final status, current status: COMMITTED, ...timeout...

3. 文件名规则与写端路由

3.1 三种文件命名（be/src/storage/lake/filenames.h）

形式命名定义

4 段 .log（有 load_id）{tablet:016X}_{txn:016X}_{loadHi:016X}_{loadLo:016X}.logfilenames.h:117

3.2 写端按 has_load_id() 路由（tablet_manager.cpp:907）

Status TabletManager::put_txn_log(const TxnLogPtr& log) {
    if (log->has_load_id()) {
        return put_txn_log(log, txn_log_location(log->tablet_id(), log->txn_id(), log->load_id())); // 4 段
    } else {
        return put_txn_log(log, txn_log_location(log->tablet_id(), log->txn_id()));                 // 2 段
    }
}

显式事务的 INSERT 走 delta_writer 时携带 load_id，所以 日志一定落到 4 段路径。

4. 根因：读端两个函数，只有一个处理 load_ids

4.1 正确实现：load_txn_log()（transactions.cpp:123）

// multi statement transaction
if (txn_info.load_ids_size() > 0) {                                  // ✅ 分支 1
    for (const auto& load_id : txn_info.load_ids()) {
        auto log_path = tablet_mgr->txn_log_location(tablet_id, txn_info.txn_id(), load_id); // 4 段
        ...
    }
} else if (!txn_info.combined_txn_log()) {                            // ✅ 分支 2
    auto log_path = tablet_mgr->txn_log_location(tablet_id, txn_info.txn_id());              // 2 段
    ...
} else {                                                              // ✅ 分支 3：combined
    ...
}

4.2 缺陷实现：publish_log_version()（transactions.cpp:507）

for (int i = 0; i < txn_infos.size(); i++) {
    if (!txn_infos[i].combined_txn_log()) {                           // ❌ 只有 2 分支
        auto txn_log_path = tablet_mgr->txn_log_location(tablet_id, txn_id);   // ❌ 强制 2 段
        auto txn_vlog_path = tablet_mgr->txn_vlog_location(tablet_id, log_version);
        auto st = fs::copy_file(txn_log_path, txn_vlog_path).status(); // → NotFound
        if (st.is_not_found()) {
            // 幂等回退：检查目标 .vlog 是否已存在
            ...
            return st;                                                 // ❌ 真正失败返回 FE
        }
    } else {
        ASSIGN_OR_RETURN(auto txn_logs, load_txn_log(tablet_mgr, {tablet_id}, txn_infos[i]));
        RETURN_IF_ERROR(tablet_mgr->put_txn_vlog(txn_logs[0][0], log_version));
    }
}

缺失 txn_info.load_ids_size() > 0 分支。当请求携带 load_ids 时，BE 按 2 段路径读取，永远命中 NotFound。

5. 为什么"正常导入"没事？

答：bug 一直存在，但平时不会被触达。FE 在 PublishVersionDaemon.publishPartition() 中按 MaterializedIndex.IndexState 将 tablet 分流： 
// fe/.../PublishVersionDaemon.java:1062-1074
for (MaterializedIndex index : indexes) {
    if (index.getState() == MaterializedIndex.IndexState.SHADOW) {
        shadowTablets.addAll(index.getTablets());   // → publishLogVersion（缺陷路径）
    } else {
        normalTablets.addAll(index.getTablets());   // → publishVersion（正确路径）
    }
}
FE 方法BE RPCBE 核心函数load_ids 分支
Utils.publishVersionpublish_versionpublish_version() → load_txn_log()✅ 有
Utils.publishLogVersion(Batch)publish_log_version(_batch)publish_log_version()❌ 没有

IndexState.SHADOW 仅在 schema change 执行期间短暂存在。alter 完成后影子索引会"提升"为 NORMAL，下一次 publish 自动走 publish_version() 的正确路径，所以：

  • 没有 alter 时：shadowTablets 永远为空，bug 路径不会被触达 → 正常。
  • 有 alter 时：显式事务 INSERT 100% 卡 COMMITTED，直到 alter 完成。
  • alter 完成后：积压事务自动追平，无数据丢失。

6. 谁会让 load_ids 非空

TxnInfoPB.load_ids 仅由 fe/.../TxnInfoHelper.java:41 写入，且条件严格：

if (state.getLoadIds() != null
        && state.getSourceType() == TransactionState.LoadJobSourceType.INSERT_STREAMING) {
    infoPB.setLoadIds(...);
}

TransactionState.loadIds 仅由 TransactionStmtExecutor.commitStmt 写入。结论：load_ids 非空 ⇔ SQL 显式事务 BEGIN; ... COMMIT;。HTTP _stream_load、HTTP 多语句事务 load 均不会触发本 bug。

7. 现场证据交叉印证（txn 18149364）

来源关键信息交叉点
BE 日志缺失文件 00000000018AF36C_000000000114EFF4.logtablet=0x18AF36C=25883500，txn=0x114EFF4=18149364
FE 日志label=019e362a-7257-70d2-b4ff-79dd422b2f8c，load_id=019e362a-725c-7fe1-8d82-07b868ff72b4，attachment=InsertTxnCommitAttachmentattachment 类型排除 HTTP _stream_load；loadIds 仅由 TransactionStmtExecutor 写入，定位为显式事务
审计日志BEGIN;(QueryId=label) → INSERT ... LEFT JOIN(QueryId=load_id) → 隐式 COMMITQueryId 与 FE/BE 日志中 label/load_id 完全一致

S3 上 4 段路径的实际文件存在；BE 却用 2 段路径查找，必然 NotFound。

8. 为什么重试救不回来

幂等回退依赖目标 .vlog 已存在；但 copy_file 的源（2 段 .log）从未存在，.vlog 永远不会被生成。每次重试都 NotFound。退出路径只有两个：

  1. Schema change 完成，影子索引升级为 NORMAL，后续 publish 走 publish_version() 正确读取 4 段 .log；
  2. Alter 被取消。

9. 影响

  • 显式事务 INSERT 在回填 alter 期间 全部卡在 COMMITTED，直到 alter 结束。
  • 客户端 txn_visible_wait_timeout 比 alter 短 → 超时返回。
  • publish 守护进程积压可能拖慢同节点上其他 stream load。
  • 无数据丢失：源 .log 实际存在，alter 结束后自动追平。

┌─────────────────────────────────────────────┐ │ FE: PublishVersionDaemon.publishPartition() │ └─────────────────────────────────────────────┘ / \ IndexState != SHADOW IndexState == SHADOW / \ Utils.publishVersion() Utils.publishLogVersion(Batch) | | BE LakeService::publish_version BE LakeService::publish_log_version(_batch) | | lake::publish_version() lake::publish_log_version() | | lake::load_txn_log() ❌ 仅 2 分支，缺 load_ids ✅ 三分支齐全 → schema change + 显式事务卡死 (load_ids/combined/默认2段)

11. 修复方向

在 publish_log_version() 中加入 txn_info.load_ids_size() > 0 分支，与 load_txn_log() 对齐。

11.1 publish_log_version 修复骨架

if (txn_infos[i].load_ids_size() > 0) {
    ASSIGN_OR_RETURN(auto per_tablet_logs,
                     load_txn_log(tablet_mgr, {tablet_id}, txn_infos[i]));
    auto& logs = per_tablet_logs[0];

    // 幂等回退：目标 .vlog 已存在则跳过
    if (logs.empty()) {
        auto vlog_path = tablet_mgr->txn_vlog_location(tablet_id, log_versions[i]);
        ASSIGN_OR_RETURN(auto fs, FileSystemFactory::CreateSharedFromString(vlog_path));
        if (fs->path_exists(vlog_path).ok()) continue;
        return Status::NotFound(...);
    }

    auto merged = merge_txn_logs(logs);                     // 复用现有合并逻辑
    RETURN_IF_ERROR(tablet_mgr->put_txn_vlog(merged, log_versions[i]));
    for (const auto& load_id : txn_infos[i].load_ids()) {
        files_to_delete.emplace_back(
            tablet_mgr->txn_log_location(tablet_id, txn_infos[i].txn_id(), load_id));
    }
    continue;
}

11.2 测试建议

  • BE UT：构造 TxnInfoPB.load_ids 多 load_id 场景，预写 4 段 .log，断言 publish_log_version 生成单个正确 .vlog 并清理源文件；覆盖幂等场景（目标 .vlog 已存在）。
  • 回归集成用例：显式事务 INSERT 与并发 ADD COLUMN ... AS COALESCE(...)，断言 txn_visible_wait_timeout 内可见，COMMITTED → VISIBLE 推进正常。

12. 风险点

  • 合并语义：多个 load_id 的 TxnLog 合并必须与 publish_version() 的 applier 一致，避免影子表与正式表数据漂移。建议复用现有 applier 路径而非重写。
  • 清理时机：失败重试需要再次读取源文件，沿用 delete_files_async 异步清理，不在主路径同步删除。
  • 代码重复：combined_txn_log 分支已复用 load_txn_log；新增 load_ids 分支建议同样复用，避免后续代码漂移。
