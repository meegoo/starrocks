1. Verdict（一句话定性）

不是用户工作负载导致；是 StarRocks 4.1 BE 内部 bug —— publish_log_version() 缺少 load_ids 分支，在表正在做需要"真实数据回填"的 Schema Change（本例为 ADD COLUMN merchantId ... AS COALESCE(...) 这种虚拟/生成列）且同时有 SQL 显式事务 INSERT 时，BE 用错误的 .log 路径查找，事务永远停在 COMMITTED，alter job 永远停在 FINISHED_REWRITING。该状态下 CANCEL ALTER TABLE 被代码强制拒绝；最终客户被迫走 DROP TABLE FORCE 路径恢复，导致停机窗口内未变可见的写入丢失。

    — 说明：故障不是性能瓶颈或容量问题，是逻辑死锁。

3. Causal chain（因果链 → 客户可见症状）

  1. 触发前提：Finaloop 在生产 4.1 集群对主键表 businessEvents.businessEventActions（Primary Key 模型，按 date_trunc('month', eventTime) 月分区，6 buckets）执行 ADD COLUMN merchantId ... AS COALESCE(...)。因为是生成列，无法走 FSE v2 fast path，必须走真实数据回填，alter 进入 RUNNING 状态创建影子索引（IndexState.SHADOW）。
  2. 事务持续写入：客户的常规 stream load（实际底层是显式事务路径，InsertTxnCommitAttachment）继续向该表写入，TxnInfoPB.load_ids 非空。
  3. BE 写入：delta_writer 调用 TabletManager::put_txn_log()，因为 has_load_id() 为 true，.log 落到带 load_id 后缀的 4 段路径（tablet_manager.cpp:907）。
  4. FE 发布：PublishVersionDaemon.publishPartition 按 MaterializedIndex.IndexState 分流——normal tablets 走 publishVersion，shadow tablets 走 publishLogVersion（PublishVersionDaemon.java:1062-1083）。
  5. BE 读端 bug：BE publish_log_version() 只判 combined_txn_log，没有 load_ids_size() > 0 分支，按 2 段路径 fs::copy_file() → NotFound（transactions.cpp:507-542）。幂等回退要求目标 .vlog 已存在，但它从未被生成。
  6. 异常吞并：publishPartition 的 try 块把 publishLogVersion（shadow）和 publishVersion（normal）串行包在一起；前者抛出后，后者根本不会执行，catch 返回 false（PublishVersionDaemon.java:1081-1110）。
  7. 事务冻结：该 partition 的 publish 失败 → txn 的 commitVersion 永远不变 visibleVersion → 事务永远 COMMITTED。
  8. Alter 死锁：LakeTableSchemaChangeJob.readyToPublishVersion() 要求 commitVersion == visibleVersion + 1 才能进入 FINISHED；条件永远不成立（LakeTableSchemaChangeJob.java:805-820）→ alter 永远停在 FINISHED_REWRITING。客户在 SHOW ALTER 中观察到 FinishTime 异常为"未来时间"。
  9. 恢复被堵死：客户在 18:30 UTC 尝试 CANCEL ALTER TABLE，被 cancelImpl 在 1166 行 jobState == FINISHED_REWRITING && tableExists() 直接拒绝："Job can not be cancelled. State: FINISHED_REWRITING"。
  10. 客户可见症状：
    • Stream load 客户端在 ~90 秒后 timeout：Wait txn state timeout 89957000 us；
    • 同分区后续 stream load / INSERT 也全部卡在 COMMITTED，整张表新写入数据完全不可见；
    • SHOW ALTER TABLE COLUMN 显示 job 永远 FINISHED_REWRITING；
    • 客户产品 5.5 小时无法接收新业务事件，"a flood of complaints from our customers"。

4. Contributing factors（放大因素 / 易混淆项）

  • 对该 ALTER 安全性的事前错误指导（Kevin Chen 在线程末尾承认）：客户在事故前一周询问过该 ALTER 是否安全，被告知"background backfill, no write lock"；事实是生成列必须走数据回填，正好命中本 bug。这导致客户在错误的时间窗口（业务高峰、未做备份）执行了 ALTER。
  • 读写端路由不对称：写端 put_txn_log 按 has_load_id() 分两路；读端 load_txn_log() 三分支齐全；但 publish_log_version() 只两分支——同一份"路由协议"在三个函数里各写一遍，导致漂移。
  • 异常打包：publishPartition 把 shadow + normal 两个 publish 顺序放在同一 try 块，shadow 抛异常时连带阻塞 normal 的可见性推进，把局部故障放大成全分区故障。
  • FINISHED_REWRITING 禁止 cancel 的设计：原本是为了保护数据完整性（这阶段已经把 shadow 推到接近完成，强行 cancel 可能丢数据），但在本 bug 场景下反而堵死了运维通道，且没有 FORCE / break-glass 旁路。
  • 幂等回退在"从未成功"场景下不可用：现有回退仅在之前成功过（.vlog 已存在）时跳过，无法识别"源文件路径就错"这一类持久性错误。
  • 误判方向：缓存：值班团队最初怀疑是缓存问题，临时禁用了 cache，浪费了 ~10 分钟排查时间。直到 Chun Zhao 注意到"one tablet block the whole ingestion"才回到正轨。
  • 客户管线混合：Finaloop 同时通过 stream load API 和 SQL 写入；表面上是 stream load 失败，但底层走显式事务路径，需要靠 InsertTxnCommitAttachment 类型字段 + loadIds 来源（仅 TransactionStmtExecutor.commitStmt 写入）才能区分。

5. Why X failed —— 为什么常规护栏没生效

  • FE 重试机制：FE 把 NotFound 当作可重试错误，但每次重试都打到同一条死路径，重试次数无上限。
  • 幂等回退（idempotency probe）：publish_log_version 在 NotFound 时探测 .vlog 是否已存在；但 .vlog 从未被生成（源 2 段 .log 根本不存在），探测必然失败。
  • Alter 的 watershedTxnId 屏障：alter 在 WAITING_TXN 阶段会等所有 txn_id < watershedTxnId 的事务完成，但本 bug 触发的事务是 alter 启动之后才提交的（txn_id > watershedTxnId），不在屏障保护范围内。
  • CANCEL ALTER TABLE 命令：默认逻辑在 FINISHED_REWRITING 阶段拒绝 cancel，没有 FORCE 选项。
  • Dashboard 告警：CPU/Mem/QPS 全部健康（资源就是没用），publish_log_version 的 WARNING 没有专用 metric 计数，必须看日志或 alter job 状态才能发现。
  • 事前安全评估：StarRocks 侧在事故前一周明确告知客户该 ALTER 安全，无任何"该路径会触发数据回填"的预警。

6. Recommended actions（按优先级排序）

Step 1 — 修根因：BE 热修复 publish_log_version()

在 be/src/storage/lake/transactions.cpp:507 的 publish_log_version() 中加入 txn_info.load_ids_size() > 0 分支，与 load_txn_log()（同文件 line 123）的三分支模式对齐：按 4 段路径读取每个 load_id 的 .log，合并写出一个 .vlog，源文件入 files_to_delete 异步清理。

// 修复骨架（在原 for 循环中作为第一个分支）
if (txn_infos[i].load_ids_size() > 0) {
    ASSIGN_OR_RETURN(auto per_tablet_logs,
                     load_txn_log(tablet_mgr, {tablet_id}, txn_infos[i]));
    auto& logs = per_tablet_logs[0];
    if (logs.empty()) {
        auto vlog_path = tablet_mgr->txn_vlog_location(tablet_id, log_versions[i]);
        ASSIGN_OR_RETURN(auto fs, FileSystemFactory::CreateSharedFromString(vlog_path));
        if (fs->path_exists(vlog_path).ok()) continue;  // 幂等
        return Status::NotFound(...);
    }
    auto merged = merge_txn_logs(logs);
    RETURN_IF_ERROR(tablet_mgr->put_txn_vlog(merged, log_versions[i]));
    for (const auto& load_id : txn_infos[i].load_ids()) {
        files_to_delete.emplace_back(
            tablet_mgr->txn_log_location(tablet_id, txn_infos[i].txn_id(), load_id));
    }
    continue;
}

预期效果：BE 二进制替换 + 滚动重启完成后，下一次 publish 调度（默认 1s 周期）就会成功生成 .vlog，分区 visibleVersion 跳到 commitVersion；LakeTableSchemaChangeJob 在再下一轮调度中进入 FINISHED，影子索引升正、alter 自然结束。对存量数据无损。

Step 2 — 客户应急路径（本次实际采用）

本次 Finaloop 在不知道存在 BE hot patch 的情况下采用了以下流程，可作为"未来类似事故、热修复未就绪"时的应急参考：

  1. 新建 businessEventActions_v2（schema 与原表一致，包含新生成列定义）；
  2. submit task re_ingest_business_events as INSERT INTO businessEventActions_v2 (...) SELECT ... FROM businessEventActions; 把当前可见数据复制到新表；
  3. DROP TABLE businessEventActions FORCE;（普通 DROP 会因 "There are still some transactions in the COMMITTED state" 被拒绝，必须 FORCE）；
  4. ALTER TABLE businessEventActions_v2 RENAME TO businessEventActions;；
  5. 重新授予该表的所有 ROLE / USER 权限（SHOW GRANTS FOR <user> → 手工 GRANT）；
  6. 重启业务摄入，等 PK index / SST 缓存预热（客户用 cache select ... from 主动 warmup）。
数据丢失警告：步骤 2 的 SELECT ... FROM businessEventActions 只能读到 visibleVersion 之下的数据。停机窗口内所有 COMMITTED 但未 VISIBLE 的写入（5.5 小时业务事件）在 DROP TABLE ... FORCE 时永久丢失。本次 Finaloop 承担了这部分数据损失。

Step 3 — 长期改进

  • 把"has_load_id() / combined_txn_log 三分支路由"抽成单一帮助函数，让所有读端（load_txn_log、publish_log_version、abort_txn 等）共享，防止下次漂移。
  • 在 publishPartition 中把 shadow 与 normal 两段 publish 解耦：shadow 失败不应阻塞 normal 的可见性推进。
  • 给 CANCEL ALTER TABLE 增加 FORCE 选项或在 FINISHED_REWRITING 状态下允许带保护条件的 cancel，避免类似 bug 把运维通道彻底堵死。
  • 新增 metric：starrocks_lake_publish_log_version_failed_total{tablet, reason}，让 dashboard 能直接报警。
  • 文档化"哪些 ALTER 走 FSE v2 fast path / 哪些需要真实数据回填"的安全矩阵，给客户和 TAM 一份明确清单——避免重复 Wednesday call 上的错误指导。

Verification（怎么确认修好了）

  • BE 日志：grep "Fail to publish log version" be.WARNING 不再有新输出，10 分钟内无新增。
  • FE 日志：grep "Fail to publish log version for tablets" fe.warn.log 同上。
  • Alter 状态：SHOW ALTER TABLE COLUMN 中受影响 job 的 State 从 FINISHED_REWRITING → FINISHED，Progress 推到 100%。
  • 事务可见性：SHOW TRANSACTION 列出的之前 COMMITTED 事务转为 VISIBLE；客户端 stream load 在 txn_visible_wait_timeout 内成功。
  • 时间预期：BE 滚动重启完成后 1–5 分钟内可见上述变化。

7. What to tell stakeholders（对外口径）

"5 月 17 日 Finaloop 生产集群（4.1 版本）出现了一次约 5.5 小时的写入不可见事故。直接触发是一次 ADD COLUMN merchantId ... AS COALESCE(...) 的虚拟/生成列 ALTER；这类 ALTER 必须走真实数据回填路径，与 StarRocks 共享存储模式下一个已确认的 BE bug（publish_log_version 缺少 load_ids 处理分支）叠加，把事务发布逻辑卡死、alter job 永远停在 FINISHED_REWRITING，且 CANCEL ALTER TABLE 在该状态下被代码直接拒绝。

事故责任与数据丢失：CelerData 此前在 Wednesday call 上错误地告知客户该 ALTER 可在不停写的情况下安全执行，未提示其会触发回填路径——这是我方失误。Finaloop 在无热修复可用的情况下被迫走 DROP TABLE FORCE + 重建路径恢复服务，停机窗口内未变可见的写入永久丢失。
修复进度：BE 缺陷修复 PR 准备中（<TBD>），将随 v4.1.<TBD> 发布；同时会 backport 到所有仍在维护的 EE 分支。
临时规避（升级前）：避免在客户业务高峰期对 PK 表执行任何需要数据回填的 ALTER（虚拟列、带 default 的 ADD COLUMN、修改 key 列等）。CelerData 将在本周内出具明确的 ALTER 安全矩阵文档。
下一步：1) Casey Luo + Luca 与 Finaloop 同步根因；2) Kevin Chen 周五前提交安全 ALTER 指南；3) CelerData 跟进 backport 与 hot patch 安排，与 Finaloop 协调升级窗口。"
